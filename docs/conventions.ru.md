# Конвенции контента

Язык контента: русский. Технические термины и названия сохраняются в оригинале (например, «dependency injection», «goroutine», «React»).

## Структура vault

```
data/
├── daily/        — автокомпилятор разговоров (Claude Code hooks → flush.py). Immutable.
├── knowledge/    — страницы из разговоров (compile.py). Не редактировать вручную.
│   ├── index.md       каталог всех knowledge-страниц
│   ├── log.md         append-only build log
│   ├── concepts/      атомарные концепты
│   ├── connections/   связи между 2+ концептами
│   ├── qa/            filed ответы на запросы
│   └── instincts/     поведенческие паттерны (trigger → action)
├── wiki/         — страницы из внешних источников (manual ingest). Опционально.
├── raw/          — исходные материалы (статьи, заметки, изображения). Immutable.
├── reports/      — lint-отчёты, agentshield-сканы. Gitignored.
└── projects.json — registry проектов (canonical + aliases)
```

## Разделение слоёв

Одно из двух главных правил системы.

- `wiki/` — внешние источники (`raw/`). Каждая страница имеет `sources:` в frontmatter. Ingest вручную.
- `knowledge/` — собственные разговоры (`daily/`). Автокомпиляция `compile.py`; руками не править.
- **Один факт — один слой.** Пересечения через `[[wiki-ссылки]]`, но страница живёт в одном.
- Граница — по происхождению, не по теме.

## Frontmatter wiki-страниц

```yaml
---
title: Название страницы
type: concept | entity | source | comparison | analysis
tags: [тег1, тег2]
sources: [имя-файла-источника.md]
created: YYYY-MM-DD
updated: YYYY-MM-DD
---
```

### Типы wiki-страниц

- **source** — краткое изложение одного источника: ключевые идеи, цитаты, оценка.
- **concept** — технология/концепция (например, «Event Sourcing», «SOLID»): определение, применение, связи.
- **entity** — сущность: инструмент, библиотека, язык, компания, персона.
- **comparison** — сравнение двух или более сущностей/подходов.
- **analysis** — авторский синтез нескольких источников.

## Ссылки

- Obsidian-style: `[[имя-файла]]` или `[[имя-файла|отображаемый текст]]`.
- Wiki-страницы без пути (Obsidian разрешает по имени).
- Raw-файлы с полным путём: `[[raw/статья.md]]`.
- Knowledge-статьи с относительным путём от `knowledge/`: `[[concepts/supabase-auth]]`.

## Изображения

`![[raw/assets/имя.png]]`. При ingest, если источник содержит изображения, упомянуть их и при необходимости просмотреть отдельно.

## Workflow

### Ingest (внешний источник)

1. Прочитать источник из `raw/`.
2. Обсудить ключевые моменты.
3. Создать страницу типа `source` в `wiki/`.
4. Обновить/создать связанные `concept` / `entity`.
5. Обновить `wiki/index.md`.
6. Добавить запись в `wiki/log.md`.

### Query

1. Прочитать `knowledge/index.md` (master).
2. Прочитать найденные страницы.
3. Синтезировать ответ с `[[ссылками]]`.
4. Если ответ ценный — предложить сохранить как `qa/`, `analysis` или `comparison`.

### Lint

Проверить:
- Противоречия между страницами.
- Устаревшие утверждения.
- Orphan-страницы (нет входящих ссылок).
- Упомянутые, но не созданные концепции.
- Пробелы в данных.

## Правила

- Никогда не модифицировать файлы в `raw/`.
- wiki → только из внешних источников. knowledge → только из разговоров. Не дублировать.
- `index.md` и `log.md` — в `knowledge/` (авто) и опционально в `wiki/` (ручной).
- При обновлении страницы — обновить `updated` в frontmatter.
- `log.md` append-only, формат: `## [YYYY-MM-DD] тип | Описание`.
- Имена файлов wiki — латиницей, kebab-case (`event-sourcing.md`, `golang.md`).
